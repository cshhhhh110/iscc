# 总要求

## 0. 入口要求

- 每个项目的 `README.md` 第一屏必须先提示“先读总规则”
- 提示内容应明确指向总目录的 `总要求.md` 和 `ACTION_LOG.md`
- 新助手进入任一项目时，应先读总目录总规则，再看项目级 README 和项目级日志

## 1. 版本命名

- 原始版本统一命名为 `original`
- 后续每次迭代必须保留独立版本号，例如 `v1.0`、`v1.1`、`v2.0`
- 版本号写在目录名中，不覆盖旧版本，不改写旧版本文件
- `模型/` 和 `提交结果/` 的主产物文件名也必须带相同版本后缀，例如 `model_bundle_v1.1.joblib`、`submission_v1.1.csv`

## 2. 保存范围

每个版本快照都必须同时保留：

- `源码/`
- `模型/`
- `提交结果/`

当前工作区只保留最新可运行版本，历史版本统一放入 `版本记录/`。

## 3. 推荐目录结构

```text
项目目录/
  源码/
  模型/
  提交结果/
  版本记录/
    original/
      源码/
      模型/
      提交结果/
    v1.0/
      源码/
      模型/
      提交结果/
```

## 4. 更新规则

- 每次完成一次有意义的迭代，都要把当前源码、模型和提交结果同步存入对应版本目录
- 不能删除原始版本
- 不能用新版本覆盖旧版本
- 版本结果文件必须和版本目录一一对应，便于回溯和复现
- 除非用户明确要求训练，不要直接启动完整 `train.py`；默认只做小规模可运行检查，确认可运行后就结束

## 5. 日志规则

- 项目内日志继续保留
- 总目录 `ACTION_LOG.md` 记录跨项目状态、版本号和关键结果
- 每次更新版本时，同时更新项目级日志和总行动日志

## 6. 工作区卫生

- `源码/` 只保留当前正在使用的代码；完成的一轮迭代必须完整归档到 `版本记录/<version>/源码/`
- `模型/` 和 `提交结果/` 也要一起归档，历史文件不覆盖
- `smoke_*` 和其他临时验证产物只允许短期存在，验证完就归档或清理，不要堆在根目录
- 根目录只保留当前活跃版本和最新交付件，不堆积旧代码、旧结果或杂项
