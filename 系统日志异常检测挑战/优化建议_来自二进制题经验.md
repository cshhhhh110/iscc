# 系统日志异常检测 — 优化建议（来自二进制漏洞检测题经验）

> 基于二进制漏洞检测 0.849→0.95+ 的实战经验，提炼可复用的优化策略。

---

## 一、评分权重分析（先搞清游戏规则）

| 子任务 | 权重 | 特性 |
|--------|------|------|
| F1_detect（异常检测） | 15% | 二分类，大概率接近饱和 |
| **IoU_loc（区间定位）** | **50%** | **主战场，最大杠杆** |
| F1_type（类型识别） | 35% | 10 类多分类 |

**核心结论**：IoU 占一半分数。逐行分类→合并区间的做法边界粗糙，预测区间和真实区间错一个 token 就扣 IoU。**最大单一提分杠杆是换序列标注模型。**

---

## 二、通用优化框架（跨题可复用）

### 1. 伪标签 — 最大单一杠杆

二进制漏洞检测经验：
- v1.4：无伪标签 → 0.901
- v1.5：v1.4 模型标注 56% test 数据 → 0.923
- v1.6：v1.5 ensemble 标注 91% test 数据 → 待验证（本地 +0.027）

执行方法：
```
第一轮：用当前最佳模型预测 test_data.csv（5,000 条）
       保守阈值（置信度 >0.9），取高置信度样本加入训练
第二轮：用更强的模型重标，降阈值至 0.7，扩大覆盖
```

关键细节：
- 伪标签 CSV 文件必须**版本化保存**（pseudo_train_v2.0.csv），归档时一并保留
- 每次迭代用**更强模型**重标（v1.5 ensemble 比 v1.4 单模型好）
- 降阈值比加种子更直接有效（91% 覆盖 vs 56% 覆盖 → 0.02+ 平台增益）

### 2. 多种子集成 — 零成本提分

- 3-5 个不同随机种子训练同一架构
- 种子间用**不同超参数**增加多样性（不同 embedding_dim、dropout）
- 预测时平均所有种子的概率
- 收益：+0.005~0.015，几乎零成本

### 3. 测试时增强（TTA）

- 推理时对输入做轻微变换，平均预测
- 二进制题：3 个不同字节窗口
- 日志题：可尝试 token dropout、同义词替换、句子内随机 mask
- 收益：+0.003~0.008，不改变模型

### 4. 本地指标 vs 平台指标必须对齐

- 二进制题教训：本地只算 CWE macro（0.8739），平台算 87 类宏平均（0.901），口径不同
- 务必搞清楚：本地 OOF score 的计算公式是否和平台一致？
- 如果平台公式是 `0.15*F1_detect + 0.50*IoU_loc + 0.35*F1_type`，本地验证必须复刻这个公式

## 三、本赛题针对性优化

### Phase 1：序列标注替换逐行分类（预期 +0.03~0.05）⭐

当前方法的核心问题是：
- 逐行独立分类 → 相邻同类行合并 → 区间边界不精确
- 预测区间和真实区间边界只要错位就扣 IoU

建议方案：
```
模型：BERT/RoBERTa/ModernBERT + CRF 序列标注层
标签：B-I-O 格式（B-timeout_retry, I-timeout_retry, O, ...）
损失：CRF loss + IoU-aware auxiliary loss
```

为什么用 BERT 系：
- 日志文本是自然语言变体，预训练模型有天然优势
- 序列标注直接输出精确区间，不需要后处理合并
- HuggingFace transformers + pytorch-crf 实现成本低

### Phase 2：伪标签扩充（预期 +0.01~0.02）

```
第1轮：用 baseline 标注 test_data.csv
       阈值：has_anomaly 置信度 >0.9 or <0.1
       区间：仅保留 type 置信度 >0.9 的样本
       生成 pseudo_train_v2.0.csv

第2轮：用 v2.0 模型重标，降阈值至 0.7
       生成 pseudo_train_v2.1.csv
```

### Phase 3：多模型集成（预期 +0.005~0.015）

```
5 折 CV × 3 seeds = 15 个模型的概率平均
异构配置：不同 BERT 变体（bert-base, roberta-base, modernbert）
          不同 learning rate、dropout
```

### Phase 4：IoU 感知训练（预期 +0.005~0.01）

在 CRF loss 基础上加 IoU-aware loss：
- 直接优化预测区间与真实区间的 IoU
- 让模型学习紧凑的区间边界
- 可参考 DICE loss 或 Generalized IoU loss 的思路

## 四、记住这些教训

### 训练注意事项
- **训练崩溃立刻清理残留进程**：僵尸进程占用内存导致后续训练不可用
- **特征缓存跨版本复用**：如果训练数据只有增量变化，不要每版重建所有缓存
- **长任务完成后主动通知**：不要让人干等

### 提交注意事项
- **提交前检查配置文件的 threshold**：一个异常的阈值（0.08 vs 0.5）可以毁掉所有优化
- **检查 label 分布**：提交前确认异常/正常比例是否合理

### 版本管理
- 每个版本归档：源码/提交结果/README/requirements/ACTION_LOG
- 模型文件留在工作区，归档处只写复现说明
- 伪标签 CSV 随版本归档保存
- 种子（random_state）硬编码在代码中，确保可复现
